# Ongrid 工程技术分析

> 分析对象：`/home/shiyao/ongrid`（版本 v0.8.6）
> 生成时间：2026-06-15
> 一句话定位：**一个能听懂基础设施、定位根因并直接修复的运维 AI Agent**，通过 Slack / Telegram 等 IM 渠道驱动，覆盖指标 / 日志 / 链路 / 拓扑爆炸半径 / 根因关联 / 远程执行 / 告警自动调查 / RAG 知识检索。

## 1. 总体定位与技术栈

| 维度 | 选型 |
|------|------|
| 后端语言 | Go 1.25 |
| 前端 | TypeScript + React 18 + Vite + TailwindCSS + Zustand |
| Agent / LLM 编排 | [CloudWeGo eino](https://github.com/cloudwego/eino) 0.8.7（ReAct graph kernel） |
| 模型供应商 | Anthropic / OpenAI / Gemini / DeepSeek / 智谱GLM / Kimi（统一走 OpenAI 兼容接口热路由） |
| 数据库 | MySQL 8（生产默认）/ SQLite（本地开发可选），ORM 用 GORM |
| 隧道 / 通信 | [singchia/frontier](https://github.com/singchia/frontier) + geminio（边端拨出，零入站端口） |
| 可观测性 | Prometheus + Loki + Tempo + Grafana + OpenTelemetry |
| 向量检索 / RAG | Qdrant + BGE 本地嵌入模型（`bge-small-zh-v1.5`，离线 ONNX 运行） |
| 鉴权 | JWT + Casbin（RBAC/ABAC 授权） |
| Web 搜索 | 自托管 SearXNG（零 API Key 元搜索） |
| API 定义 | Protobuf（buf 管理）/ 隧道层走 JSON over geminio |
| 许可证 | Apache 2.0 |

技术栈整体是一套**自托管、可一键部署的云原生 AIOps 平台**，强调「零入站端口、自带可观测栈、自带模型路由」。

## 2. 架构总览

### 2.1 三大有界上下文（Bounded Context）

代码以 monorepo 组织，`internal/` 下划分三个互不直接 import 的 BC：

```
internal/
├── iam/         身份与访问管理（user 子域；org/membership 已在 ADR-005 中裁剪）
├── manager/     云端核心：AIOps、告警、设备、拓扑、知识库、监控、IM 桥、审计、报表……
└── edgeagent/   边端代理：采集器、插件、bash 沙箱、host_files、webshell、skill 执行
```

公共能力沉淀在 `internal/pkg/`（auth、config、llm、prom、tracing、qdrantx、embedding、tunnel 等），**不允许反向依赖任何 BC**。

### 2.2 分层与依赖约束（强制）

每个 BC 内部遵循严格分层，由 `.go-arch-lint.yml` 通过 `make arch-lint` 在 CI 强制校验：

```
cmd → server → service → biz → data → model
                            ↑
                       shared_pkg / api
```

关键红线（见 `AGENTS.md`）：
- 三大 BC 两两禁止互相 import，只能通过 API / 事件 / `internal/pkg/` 通信；
- `service` 层禁止越过 `biz` 直接访问 `data`；
- 接口在消费方定义，禁止循环依赖；
- 依赖通过构造函数注入，禁止全局可变变量。

这是一套相当工程化的「单服务 + 清晰分层 + 架构 lint 守门」的约束，遵循 [gospec](https://github.com/singchia/gospec) Go 后端 SDLC 规范。

### 2.3 两个二进制

| 二进制 | 位置 | 角色 |
|--------|------|------|
| `ongrid` | `cmd/ongrid/main.go`（~3000 行 composition root） | 云端 manager：组合 iam + manager 两个 BC，暴露公开 HTTP API、`/metrics`，并作为 service 端拨上游 frontier broker |
| `ongrid-edge` | `cmd/ongrid-edge/main.go` | 边端代理：拨出隧道、上报主机指标、提供工具 RPC handler（get_host_load 等） |

## 3. 隧道通信模型（零入站端口，ADR-007）

这是 Ongrid 最具特色的设计之一：

```
   边端主机                  上游 frontier broker                 云端 manager
 ┌────────────┐   拨出     ┌──────────────────────┐    拨上     ┌─────────────┐
 │ ongrid-edge│ ────────► │ singchia/frontier     │ ◄───────── │  ongrid     │
 │            │ :40012    │ (geminio 终结)         │  :40011    │  (service端) │
 └────────────┘ edge-bound└──────────────────────┘ service端  └─────────────┘
```

- **边端只拨出（dial out）**，主机上无需开放 22 / 80 / 443，没有 jumpbox；
- frontier broker 终结 geminio 隧道，边端在 `:40012` 注册，manager 在 `:40011` 建立长连接；
- manager 注册生命周期回调（`GetEdgeID` / `EdgeOnline` / `EdgeOffline`）与反向调用 handler（`register_edge` / `heartbeat` / `push_host_metrics`）；
- manager 发起的工具调用（如 aiops 工具）经同一 SDK 通过 `frontierbound.Client.Call` 反向打到指定边端；
- 隧道消息 MVP 阶段以 **JSON over geminio** 编码（非 gRPC），proto 仅定义消息体形状（`api/tunnel/v1/tunnel.proto`），刻意自包含、不跨包 import。

衍生能力：**浏览器 SSH**（反向隧道 webshell，无需密钥 / 跳板机，全程审计）。

## 4. AIOps Agent 内核（核心亮点）

代码集中在 `internal/manager/biz/aiops/`，子模块：`agent / chatruntime / graph / investigator / mentions / tools / toolreplay`。

### 4.1 双内核：graph vs legacy

由 `ONGRID_AGENT_KERNEL` 切换（默认 `graph`，可切 `legacy` 做二分排障）：
- **graph**：基于 eino 的 ReAct graph kernel（`graph/react.go`），包装 `MessageAssembler → ReActSubgraph → OutputProjector`，复用 eino 标准节点名（ChatModel / Tools）以便审计 / SSE / 持久化 callback 按节点过滤；
- **legacy**：历史 for-loop runner，所有工具始终全 schema 暴露。

### 4.2 Coordinator + Specialist 多智能体

`agents/` 目录用 Markdown + frontmatter 定义角色（persona），coordinator 把问题分派给专家子 agent：

| Agent | 职责 |
|-------|------|
| `specialist-sre` | SRE / 可观测性：告警优先级、黄金四信号、SLO、错误预算、异常趋势 |
| `specialist-compute` | 计算资源 |
| `specialist-disk` | 磁盘 / 文件系统底层排查 |
| `specialist-network` | 网络层（OVS / iptables / netns） |
| `specialist-ops` | 单机操作 / 服务启停 / 配置改动 |
| `incident-investigator` | 单条 incident 端到端走 metric + log + trace 的关联诊断 |
| `reporter` | 报表生成 |
| `reviewer` | 复核 |

每个 persona 声明 `tools`（可用工具白名单）、`permission_mode`（如 read-only）、`max_turns`，并在 prompt 中规约工作流（如 SRE 专家强制「先查 KB → 黄金四信号语言 → 给优先级 → 派活」）。

### 4.3 工具体系（aiops/tools）

约 30+ 工具，覆盖：
- **查询类**：`query_promql` / `query_logql` / `query_traceql` / `query_edges` / `query_incidents` / `query_alert_rules` / `query_change_events`
- **拓扑类**：`get_topology` / `expand_topology` / `find_topology_node`
- **关联诊断**：`correlate_incident` / `get_incident_detail` / `get_active_incidents`
- **集群分析**：`find_outlier_edges`（找异常的那一台）/ `rank_edges` / `get_edge_summary`
- **执行类**：`bash`（沙箱）/ `restart_service` / `host_files` / `host_processes`
- **数据库**：`analyze_database_status`
- **元工具**：`AgentTool`（派子 agent）/ `SendMessage` / `ToolSearch`（延迟加载） / `query_knowledge`（RAG）

### 4.4 ToolBag 延迟加载（prompt 预算优化）

`tools/toolbag.go` 实现了类 Anthropic「ToolSearch」的延迟工具加载：
- 工具分两层：**core**（始终全 schema，约 15 个）+ **specialty**（按需）；
- 当工具总数 > 阈值（`ONGRID_TOOLBAG_DEFERRAL_THRESHOLD`，默认 30），specialty 工具只暴露 `(name, description, when_to_use)`，LLM 须调 `ToolSearch` 拉取真实 schema 后才能调用；
- 目的：随 marketplace 增长工具数膨胀时，控制每次 ChatModel 调用的 JSON Schema token 成本。

### 4.5 告警自动调查（investigator）

`biz/alert/investigator` + `biz/aiops/investigator`：告警触发后自动 spawn 一个 RCA worker，走拓扑、关联 m/l/t、把「为什么」定位到源码行，并把根因写回聊天渠道。

### 4.6 多模型热路由

`internal/pkg/llm/router.go` 的 `MultiClient` 按 `ChatReq.Provider` 分发到预建子客户端。由于 Anthropic / 智谱 / Gemini / OpenAI 都提供 OpenAI 兼容 chat completions 接口，所以统一按 `(apiKey, baseURL)` 路由，SDK 保持一致。配套 `budget.go` 做预算控制、`metrics.go` 做指标埋点。

## 5. 边端代理（edgeagent）

`internal/edgeagent/` 结构：
- **collector / biz/collector**：主机指标采集；
- **plugins/**：可插拔遥测插件 —— `hostmetrics` / `procmetrics` / `databasemetrics` / `custommetrics` / `metrics` / `logs` / `traces`，对应集成 node_exporter / process-exporter / 各类 DB exporter / promtail / otelcol（由 Makefile 的 `fetch-*` target 在打包时下载）；
- **bash**：受限 bash 沙箱（只读 + 26+ 巡检工具，每次调用审计）；
- **host_files / restart_service / webshell**：受控的文件读取、服务重启、反向 shell；
- **skill**：execute_skill RPC 分发器，路由到 `internal/skill` 注册表。

数据面遥测（指标 / 日志 / 链路）由边端直接推往云端 manager 暴露的端点（ADR-014 / ADR-015），manager 把 endpoint 作为数据面地址下发给边端。

## 6. Skill 系统

两套 skill 概念：
- **`internal/skill/`**：通用 skill 框架（loader / registry / schema / subprocess 执行），内置 builtin：`probe_tcp` / `probe_http` / `probe_dns` / `read_journal` / `tail_file` / `host_netns_inspect` / `web_search` / `restart_service`；
- **`skills/`（仓库根）**：以 `SKILL.md` 声明的 skill 包（`bash` / `host-files` / `restart-service`），支持 `activation:keyword` —— graph kernel 下，带激活关键词的 skill 不进 prompt，除非用户 query 命中关键词（`SkillRegistry.Resolve` 过滤）；
- **marketplace**（`biz/marketplace`）：skill 市场，可扩展工具集。

## 7. 前端（web/）

- 技术：React 18 + Vite + Tailwind + Zustand（状态）+ React Router；图表用 recharts，拓扑图用 `@xyflow/react` + dagre，终端用 xterm.js，Markdown 用 react-markdown。
- 页面（`web/src/pages/`）：Dashboard、Agents、Alerts / AlertRules、ChatThread、Edges / EdgeDetail、Topology、Monitor、Logs、Traces、Knowledge / KnowledgeRepos、Incidents、Reports / ReportSchedules、Skills / SkillRun、DeviceShell（浏览器 SSH）、Settings、Login。
- 设计规范（`AGENTS.md` 前端 UI 段）极其细致：「跟随大多数页面」、强制复用 `components/ui/`、zinc 中性骨架 + indigo 主操作 + 语义色（emerald/amber/red/sky）、克制美学（禁 animate-pulse / 发光 / hover:scale）、light/dark 双主题、i18n 走 `tr('中文','English')`、视觉改动必须 headless 截图验证。

## 8. 数据与存储

- **MySQL 8** 为生产默认后端，SQLite 为本地开发可选（`data` 目录从 `mysql/` 重命名为 `sqlite/`，ADR-005）；
- GORM 作 ORM，Casbin 用 gorm-adapter 持久化策略；
- 每个 manager 子域都有 `model`（领域模型）/ `data`（仓储实现，按存储引擎分子目录）的完整分层；
- 向量数据走 Qdrant（`internal/pkg/qdrantx`），嵌入走本地 BGE ONNX 模型（`internal/pkg/embedding`，fastembed-go + onnxruntime）。
- 规范红线：SQL 全参数化、PII 加密存储、多租户接口强制 `tenant_id` 过滤、Redis key 必设 TTL。

---

## 9. 可观测性

- 自带完整 LGTM 栈：Prometheus（remote_write 接收 + PromQL 查询后端）、Loki（日志）、Tempo（链路）、Grafana（嵌入式，manager 首启自动 bootstrap Service Account token）；
- manager 侧 Prometheus（ADR-009）接收开放集样本，aiops `query_promql` 工具从中读；
- 强制规范：对外服务暴露 `/healthz` `/readyz` `/metrics`；日志结构化（slog）+ `trace_id`；高基数字段（user_id / email / url）禁作 Prometheus label；OpenTelemetry 链路埋点（`internal/pkg/tracing`）。

## 10. 部署与发布

- **一键自托管**：`deploy/install/install.sh`，支持 Ubuntu 22.04+ / Debian 12+ / RHEL/Rocky 9，amd64 / arm64；
- **生产形态**（ADR-008）：nginx 终结 TLS（443/80）+ 托管 SPA + 反代 `/api`，HTTP 8080 不再暴露到宿主；可观测后端端口全部内网化，公网访问经 nginx `auth_request` → manager edgeauth 鉴权；
- **本地开发**：`deploy/docker-compose.yml`，一把拉起 mysql / ongrid / nginx / frontier / prometheus / loki / tempo / searxng / qdrant / grafana 共 10 个服务；
- **发布打包**：`make package` / `package-all` 交叉编译 edge 四目标（linux/darwin × amd64/arm64）、拉取各 exporter、构建镜像、预拉离线嵌入模型，产出单架构 / 双架构 release tarball；
- **边端升级**（ADR-024）：edge upgrade bundle 机制（`build-edge-bundle`，manager 读 .sha256 触发升级，nginx 提供 bytes）；
- **CI**：`.github/workflows/release.yml`（发布流水线）；规范要求 CI 强制 `-race` + `govulncheck` + 依赖/镜像漏扫。

## 11. 工程质量与规范

项目工程化程度很高：
- **gospec SDLC 全流程规范**（AGENTS.md 路由到 spec 子文件），覆盖编码 / 架构 / API / 数据 / 测试 / CI / 安全 / 运维红线；
- **架构 lint** 守边界（go-arch-lint）；**golangci-lint** 守代码（`.golangci.yml`）；
- **测试分层**：单元测试（大量 `*_test.go`）、`test-race`、`test-integration`（testcontainers）、`test-e2e`（默认 fakes，`test-e2e-live` 打通真实外部服务）；
- **API 优先**：所有 API 变更先改 `.proto`，禁改生成代码；Handler 强制 Swagger 注释；响应统一 `{code, message, data}`；破坏性变更走新版本；
- **Conventional Commits** + 禁 force push main + 禁提交密钥；
- **ADR 驱动**：代码注释大量引用 ADR 编号（ADR-001/005/007/008/009/012/013/014/015/024），决策有据可查；
- 多语言 README（9 种语言）+ 中英双语 ROADMAP + CONTRIBUTING / SECURITY / CODE_OF_CONDUCT 齐全。

## 12. 总结：技术亮点与潜在关注点

### 亮点
1. **零入站端口的边端拨出隧道**（frontier/geminio）—— 安全模型干净，落地运维友好；
2. **Coordinator + Specialist 多 agent + ToolBag 延迟加载** —— 在真实 prompt 预算约束下做工程化 agent 编排，而非简单 ReAct loop；
3. **自带完整 LGTM + RAG 栈**，agent 自己写查询，一键部署即可用；
4. **多模型热路由 + 预算控制**，不绑死单一供应商；
5. **极强的工程规范约束**（架构 lint、分层红线、ADR、测试分层），代码可维护性高；
6. **告警驱动的自动 RCA** 把根因定位到源码行，闭环到 IM 渠道。

### 潜在关注点
1. `cmd/ongrid/main.go` 约 3000 行的 composition root，组合逻辑集中，后续依赖增长时可读性 / 测试性需关注；
2. 隧道层 MVP 用 JSON over geminio，proto 仅描述形状未做二进制 —— 高吞吐场景下序列化成本与版本演进需评估（ADR 已标注 Phase 2 可能切 proto binary）；
3. 自带 9~10 个依赖服务的部署，资源占用与运维复杂度对小团队有一定门槛；
4. 本地 BGE 嵌入依赖 ONNX runtime（`libonnxruntime.so`），跨平台 / 边缘资源受限场景需注意；
5. agent 具备 bash / 服务重启 / 文件操作等**写能力**，虽有 read-only persona、沙箱与全程审计，仍需严格的权限模式与多租户隔离保障（规范已覆盖，落地需持续验证）。
